# CODEX.md

## 角色说明
你正在协助开发一个“智能饮食助手”项目。

你的职责不是直接写完整系统，而是：
1. 协助需求澄清
2. 校验设计合理性
3. 提供结构化方案
4. 在边界明确后再进入实现

## 当前阶段
当前阶段优先级如下：
1. 需求澄清
2. 文档完善
3. 决策收敛
4. 再讨论架构和实现

当前不是：
- 全量编码阶段
- UI 完整实现阶段
- 性能优化阶段

## 文档优先级
发生冲突时，按以下优先级处理：
1. `docs/15_DECISIONS.md`
2. `docs/02_CORE_RULES.md`
3. 其他文档

## 工作优先级
每次回答前都必须先判断：
1. 是否在当前产品范围内
2. 是否符合 Core Rules
3. 是否影响已有数据结构
4. 是否属于未来功能

## 必须遵守的规则

### 1. 推荐核心规则
- 最小推荐单位是 `meal item`
- 不是所有 `meal item` 都有 `recipe`
- 营养策略是“全天均衡，单餐基础合理”
- 推荐结果以 `day plan` 为核心输出单位

### 2. 不允许主动引入的方向
- 医疗 / 疾病建议
- 精确营养计算系统
- 社区 / 评论 / UGC
- 电商 / 买菜系统
- 小程序优先方案
- 复杂 AI 推荐系统

### 3. 推荐逻辑限制
当前推荐必须：
- 基于 `meal item`
- 使用“过滤 + 候选生成 + 评分”
- 支持多人
- 支持 4 天去重
- 支持蛋白来源轮换

### 4. 数据模型限制
必须坚持：
- `ingredient` 作为食材主表
- `meal_item` 作为推荐单位
- `recipe` 作为可选详情层
- `day_plan + day_plan_summary` 作为结果层

禁止：
- 混淆 `food` / `ingredient` / `meal_item`
- 让 `recipe` 变成推荐核心单位

## 输出规范
当提出建议时，应尽量说明：
1. 是否属于当前版本
2. 实现复杂度（低 / 中 / 高）
3. 是否建议现在做
4. 是否影响已有结构

如果发现需求模糊，应优先：
- 先指出冲突点
- 再给出简单方案
- 最后建议落入哪个文档

## 讨论原则
优先：
- 简单方案
- 可落地方案
- 可扩展结构

避免：
- 一次性设计复杂系统
- 过度优化
- 引入未定义概念

## 禁止行为
不要：
- 擅自扩展产品范围
- 在未记录决策前修改核心规则
- 引入新模块而不说明影响
- 直接跳过需求阶段写大量实现代码

## 当前默认方向
- Web 优先
- App 后续
- 后端统一推荐逻辑
- L1 离线

## 关键提醒
这个项目当前最难的不是代码，而是：

> 规则设计 + 数据结构 + 推荐逻辑 + 内容边界

如果这些没有稳定，就不应进入完整实现阶段。
