# 15 Decisions

## 目的
本文件记录已经明确确认的产品决策，用于解决跨文档冲突。

本文档优先级最高。

## 使用规则
- 已经确认的事项，写入“已确认决策”
- 仍在讨论中的事项，写入“待确认事项”
- 若其他文档与本文件冲突，以本文件为准

## 已确认决策

### D-001 产品定位
确认日期：2026-03-17

项目定位为“健康饮食助手”，不是：
- 医疗系统
- 疾病管理系统
- 营养师处方工具
- 社区平台
- 买菜电商平台

### D-002 首版目标
确认日期：2026-03-17

首版目标是先做通一条完整主链路：
- 输入人数
- 获取今日三餐
- 查看可执行内容
- 查看今日购物清单
- 查看基础食物与营养内容

### D-003 推荐核心单位
确认日期：2026-03-17

推荐最小单位是 `meal item`，不是 `recipe`。

结论：
- 推荐逻辑围绕 `meal item`
- `recipe` 只是详情层
- 不是所有 meal item 都需要 recipe

### D-004 结果核心单位
确认日期：2026-03-17

面向用户的核心结果单位是 `day plan`。

结论：
- 推荐以整天为单位输出
- 刷新按整天处理
- 购物清单和营养汇总依附于当天 day plan

### D-005 营养策略
确认日期：2026-03-17

营养策略确定为：
- 全天均衡为主
- 单餐基础合理即可
- 首版不追求医疗级精度

### D-006 早餐策略
确认日期：2026-03-17

早餐采用预定义 `breakfast_combo`。

结论：
- 首版优先做 10~15 个高频早餐组合
- 早餐不按菜数变化
- 早餐主要通过总量适配人数
- 早餐在排除食材场景下仍只从 `breakfast_combo` 中选择
- 若候选不足，优先放宽早餐多样性偏好，再使用低重叠早餐兜底组合
- 若仍不足，应明确提示“当前排除条件下早餐可选较少”

### D-007 多人支持方式
确认日期：2026-03-17

多人支持通过“增加菜数 + 调整克重”实现，而不是所有菜品简单乘人数。

结论：
- 一餐最多 4 个 meal items
- soup 每餐建议最多 1 个
- food 在午晚餐中只作为辅助项

### D-008 购物清单范围
确认日期：2026-03-17

购物清单首版只覆盖：
- 今日三餐
- 原材料
- 跨餐合并后的食材数量

明确不包括：
- 调味料
- 多天采购
- 库存扣减
- 智能替代

补充结论：
- 除调味料外，可直接购买和食用的非调味项也进入购物清单

### D-009 内容模块定位
确认日期：2026-03-17

`Food Library` 与 `Nutrition Knowledge` 在首版中属于辅助理解模块，不是主推荐引擎。

首版规模：
- Food Library：30~50 个高频食物
- Nutrition Knowledge：10~20 个高频营养素

### D-010 离线范围
确认日期：2026-03-17

首版只做 L1 离线。

结论：
- 支持离线查看最近内容
- 不支持离线生成新三餐

### D-011 平台优先级
确认日期：2026-03-17

平台优先级为：
1. Web
2. Mobile App
3. Mini Program（未来再考虑）

### D-012 首版复杂度控制
确认日期：2026-03-17

首版明确不做：
- 复杂 AI 推荐系统
- 账号系统
- 收藏系统
- 社区 / 评论 / UGC
- 医疗建议
- 疾病饮食管理

### D-013 首版输入边界
确认日期：2026-03-17

首版必须支持的核心输入：
- `people_count`

当前结论：
- `disliked_ingredient_ids` 作为首版可选输入
- `goal` 保留为后续扩展字段，但不作为首版必须完成项
- `date` 不作为首版生成接口输入，只用于历史记录查询

### D-014 首版人数上限
确认日期：2026-03-17

首版支持人数范围为 1~4 人。

结论：
- `people_count` 超出 4 不属于首版支持范围
- 一餐最多 4 个 meal items
- 首版多人适配优先覆盖家庭小规模场景
- 推荐菜数建议区间：
  - 1 人：1~2 个 meal items
  - 2 人：2~3 个 meal items
  - 3~4 人：3~4 个 meal items

### D-015 基础不吃食材过滤
确认日期：2026-03-17

首版支持基础 ingredient 级排除。

结论：
- 用户可提供可选的 `disliked_ingredient_ids`
- 最多支持 3 个排除 ingredient
- 系统在推荐前过滤包含这些 ingredient 的 meal items
- 该条件优先级高于去重和蛋白轮换等次级规则
- 首版不做智能替代、类别级排除、过敏级保障
- 若候选不足，优先放宽次级规则和菜数下限，但不放宽明确排除条件

### D-016 推荐理由
确认日期：2026-03-17

首版为整天推荐结果提供简短推荐理由。

结论：
- 推荐理由以 1~3 条简单说明为主
- 理由基于规则命中，不做复杂评分展开
- 理由应保持中性、简洁、不医疗化

### D-017 历史 day plan 浏览
确认日期：2026-03-17

首版支持历史 day plan 浏览。

结论：
- 用户可查看最近 7 天的 day plan 记录
- 历史详情可包含三餐结果、推荐理由、购物清单快照和营养汇总
- 该能力以“回看和参考”为主
- 首版不支持历史记录恢复、复制、编辑和重新生成
- 同一天若发生多次刷新，只保留当天最后一次结果

### D-018 刷新计数口径
确认日期：2026-03-17

`refresh_count` 按北京时间自然日计算。

结论：
- 每个自然日单独计算刷新次数
- 每天 00:00~23:59 视为同一天
- 每个自然日最多刷新 3 次
- 同一天内即使修改 `people_count` 或 `disliked_ingredient_ids`，也共用当天刷新额度
- 同一天只保留页面最终停留的最后一次结果
- 达到当天刷新上限后，直接禁止继续刷新，并展示当天最后保存结果
- 不同日期之间的刷新次数互不影响

### D-019 recipe 详情边界
确认日期：2026-03-17

首版不要求每个 meal item 都有 recipe。

结论：
- `breakfast_combo` 默认不要求正式 recipe
- `dish` 和 `soup` 可按内容需要配置 recipe
- 只要 `has_recipe = true`，就必须提供步骤图
- 首版不把时间、难度、食材数、特殊厨具、复杂工序设为硬性产品门槛

### D-020 day_plan_summary 统计口径
确认日期：2026-03-17

`day_plan_summary` 中的计数字段由后端按克重自动换算为份数。

结论：
- `vegetable_count` 由后端按克重自动换算为份数
- `fruit_count` 由后端按克重自动换算为份数
- 不按食材种类数累计

### D-021 meal_item / recipe 录入节奏
确认日期：2026-03-17

首版 `meal_item` 和 `recipe` 采用逐步录入。

结论：
- 当前不设硬性固定数量门槛
- 以高频、可执行内容优先
- 后续可根据推荐稳定性再补数量目标

### D-022 日期边界
确认日期：2026-03-17

首版只支持生成“今天”的 day plan。

结论：
- 不支持直接生成明天或其他未来日期计划
- `GET /api/day-plan/today` 只面向当天生成
- `date` 只用于历史记录读取，不作为首版生成输入

### D-023 单位换算字段
确认日期：2026-03-17

首版通过统一字段控制后端计算、前端展示和份数换算。

结论：
- `backend_unit`：后端存储与计算单位
- `frontend_unit`：前端展示单位
- `serving_size_backend_amount`：单份对应的后端单位数量
- `backend_to_frontend_ratio`：后端单位换算到前端单位的比例

### D-024 首版字段字典
确认日期：2026-03-17

首版枚举类字段以 `docs/04_DATA_MODEL.md` 中的字段字典为准。

结论：
- 枚举类字段不应在录入时自由扩展
- 多值标签字段统一使用 `|` 连接
- 若后续需要新增枚举值，应先更新决策与数据模型文档

### D-025 不吃食材交互方式
确认日期：2026-03-17

首版“不吃食材”通过当前 ingredient 列表搜索并选择。

结论：
- 前端应基于现有 ingredient 列表完成搜索和选择
- 首版不支持自由文本输入排除食材
- 若当前食材库无匹配项，应明确提示暂不支持该食材筛选

### D-026 极端失败处理
确认日期：2026-03-17

若当前条件下无法生成完整且合规的新方案，系统应明确失败而不是伪造结果。

结论：
- 不得违反用户明确排除条件
- 不得用不完整新方案覆盖当天已有成功方案
- 若当天已有成功生成结果，应保留原方案并提示当前新条件未生效
- 若当天尚无成功生成结果，应返回明确失败状态，不保存新的 day plan

### D-027 系统提示语义
确认日期：2026-03-17

首版关键提示应统一口径。

结论：
- 刷新次数用尽时，必须明确提示“刚才修改的条件未生效”
- 候选不足时，应明确提示系统已尽量保留可用组合
- 无法生成新方案时，应明确提示是否保留了当天原方案
- 具体推荐文案以 `docs/14_CONTENT_RULES.md` 为准

### D-028 第一阶段完成标准
确认日期：2026-03-17

第一阶段以“需求收敛完成”为目标。

完成标准：
- 产品范围已稳定并写入 `docs/01_PRODUCT_SCOPE.md`
- 核心规则已稳定并写入 `docs/02_CORE_RULES.md`
- 关键决策已同步写入 `docs/15_DECISIONS.md`
- 数据模型主字段与首版字段字典已明确
- 关键异常路径已明确，包括刷新上限与无法生成新方案
- 系统提示语义已统一
- 可以进入第二阶段的数据规范与数据准备工作
